What Every Server Should Be Able To Do

Posted on 1970-01-01 by preaction

When I develop software, there is a minimum level of functionality that must be provided by the OS for me to do my job. Frequently, this functionality is disabled or destroyed by the system administrators, making my job a lot harder, since I am forced to develop workarounds that are inevitably more bug-prone, slower, and less robust than the solutions that already exist.

Just this week I was forced to write an e-mail queueing system that jumps in when our mail relay is down, but only for certain e-mails because that's the only set of e-mails I could hook into. A local smtpd that only responded to localhost would have provided me this functionality for every e-mail with no change to my application...

If you are a sysadmin of a server I develop on, your system should be able to:

  • Relay mail from localhost
  • Run tasks on a schedule (cron)
  • Access to logs for the above (mail, cron)
  • Resolve the DNS I need (likely the whole Internet)
  • Resolve to DNS
  • Log in using plain text (ssh)
  • Persist my login sessions (tmux/screen)
  • Access my dev toolbox (vim, git, and github.com)

Every single server is going to need these things. No server should be without any of them. I especially should not have to track someone down at 8:00 PM to look at a log because one my cron jobs failed to run (and somehow failed to send e-mail) and clients are screaming at me.
